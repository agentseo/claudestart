# Claude Code — Зачем это всё

Объяснение для тех, кто только знакомится с Claude Code и хочет понять почему окружение устроено именно так.

Инструкция по установке — в `setup.md`. Её можно отдать Claude, и он настроит всё сам.

---

## Что именно настраивается

`setup.md` делает следующее:

**Функция запуска** — `cc` для быстрого старта Claude без подтверждения каждого действия. Один аккаунт — одна функция (второй аккаунт можно добавить опционально).

**Доступ к Anthropic** (опционально) — если из твоего региона `api.anthropic.com` заблокирован, запуск заворачивается через свой канал (прокси / SSH-тоннель / VPN). Способ выбираешь сам.

**RTK** — прокси для экономии токенов. После установки работает автоматически, ничего менять не нужно.

**Headroom** (опционально) — прослойка, сжимающая контекст, чтобы в окно влезало больше; для больших и долгих сессий.

**Хуки в Claude Code:**

- при завершении ответа — звуковой сигнал и красная вкладка в tmux
- при записи файла — авто-форматирование (prettier, ruff, rustfmt, gofmt в зависимости от языка)
- перед каждой командой — RTK перехватывает и оптимизирует

**Статус-строка** внизу Claude Code — показывает какой аккаунт активен, модель, текущую ветку git (со звёздочкой если есть несохранённые изменения), заполненность контекста и rate limits.

**Список разрешений** — Claude может без вопросов выполнять стандартные команды (`git`, `docker`, `ls`, `python` и др.). `rm` намеренно исключён — требует подтверждения.

**tmux** — конфиг с удобными шорткатами, темой Catppuccin, сохранением сессий между перезагрузками (tmux-resurrect) и автосохранением каждые 15 минут.

## Claude Code — это не чат

Claude.ai (веб) — это диалог. Claude Code — это агент в терминале, который реально работает с твоей машиной: читает и пишет файлы, запускает команды, работает с git, docker, браузером. Разница примерно как между "попросить кого-то помочь по телефону" и "пустить человека за компьютер".

Это меняет сценарий использования: вместо "объясни мне как написать этот компонент" — "напиши этот компонент, запусти тесты, исправь ошибки, задеплой". Claude Code делает всё сам.

## Windows: почему WSL

Claude Code — Linux/macOS инструмент, на чистом Windows не работает. WSL (Windows Subsystem for Linux) даёт полноценный Linux прямо внутри Windows без отдельной виртуалки.

Из WSL можно:

- вызывать Windows-программы: `powershell.exe`, `explorer.exe`, `code.exe`
- обращаться к файлам Windows через `/mnt/c/...`

Из Windows можно обращаться к файлам WSL:

- в проводнике: адресная строка → `\\wsl$\Ubuntu` (или `\\wsl.localhost\Ubuntu`)
- папка домашнего каталога: `\\wsl$\Ubuntu\home\<username>`
- удобно добавить в "Быстрый доступ" проводника — правой кнопкой → "Закрепить на панели быстрого доступа"
- VS Code: запусти `code .` из WSL-терминала — откроется с автоматическим WSL Remote подключением

## Docker

В этом конфиге Docker используется как инструмент разработки — Claude работает с контейнерами через команды `docker *` и `docker-compose *`. Сам Claude Code запускается в WSL, не в контейнере.

Альтернативный сценарий — запустить Claude Code внутри Docker-контейнера для полной изоляции — здесь не используется.

## Почему tmux

В WSL2 все терминалы делят **одну виртуальную машину и одно ядро Linux** — открыть ещё одно окно не создаёт отдельный экземпляр Linux-окружения, это просто ещё один шелл в той же VM. Так что дело не в «дублировании Linux». Память съедает не терминал сам по себе, а то, что в нём запущено: общая VM (`vmmem`) плюс твои процессы — каждая сессия Claude это `node` и её MCP-серверы (а с Playwright ещё и Chromium). Когда таких сессий много, набором отдельных окон терминала ими тяжело управлять, и легко потерять работу, случайно закрыв окно.

Tmux решает именно это — собирает всё в один мультиплексор с сохранением состояния (память он сам по себе почти не экономит, выигрыш в другом):

- **Сессии не умирают** — закрыл терминал или отвалился VS Code, открыл снова — всё на месте (`tmux attach`); процессы внутри tmux продолжают жить
- **Несколько панелей** — Claude в одной, ты работаешь в другой, рядом логи
- **Цвет вкладки как уведомление** — вкладка tmux меняет цвет по состоянию агента: 🔵 синий — работает, 🔴 красный — закончил ход (твой ход), 🟡 жёлтый — ждёт пермишена. Видно, даже если смотришь на другое окно

**Запускать удобнее всего внутри VS Code** (WSL-remote режим): интегрированный
терминал авто-аттачится к сессии 0, и Ctrl+Click по путям из ответов Claude
открывает файлы вкладками рядом. Отдельный Windows Terminal не нужен.
Настройка — в `setup.md` §12.

## Почему RTK

Claude Code платный — каждый токен стоит денег. Стандартные инструменты выдают избыточный вывод: `git log` без параметров возвращает тысячи строк, Claude их читает, ты за них платишь.

RTK (Rust Token Killer) — прокси, который перехватывает команды Claude и заменяет их на компактные эквиваленты. `git log` → `rtk git log` (только нужные поля). Прозрачно, без изменения поведения, экономия 60–90% на типичных операциях.

Работает через хук `PreToolUse`: каждая bash-команда Claude автоматически проходит через RTK.

## Если Anthropic заблокирован в твоём регионе

Claude Code обращается к `api.anthropic.com`. Если из твоей страны этот адрес недоступен, запуск можно направить через свой канал — прокси, SSH-тоннель или VPN до разрешённого региона. Какой именно способ — твой выбор; `setup.md` (§2.1) даёт общую схему, а не готовый чужой конфиг. Удобно завернуть это в обёртку, чтобы команда запуска сама поднимала канал, и ты работал с Anthropic напрямую через него.

---

## Headroom — когда контекста не хватает

Headroom — необязательная прослойка между Claude и API: на лету сжимает то, что забивает контекст (длинные выводы команд, большие файлы). В окно влезает больше, и сессию можно дольше не чистить. Ставится отдельно и запускается своей функцией (`cch`) рядом с обычной `cc`. Детали — `setup.md` §2.2.

---

## Запуск: одна функция `cc`

Claude Code привязан к аккаунту Anthropic. В базовом сетапе — один аккаунт и одна функция запуска `cc` (конфиг в `~/.claude/`).

Если нужно держать изолированно два аккаунта (например личный и рабочий) — это опция: заводится второй `CLAUDE_CONFIG_DIR` (`~/.claude-work/`) и вторая функция. У каждого аккаунта свои плагины, история и лимиты, конфиги не наследуются. Деталь — в `setup.md` §8.

## GitHub CLI (gh)

Claude Code умеет работать с GitHub — создавать PR, смотреть issues, проверять статус CI. Но для этого нужен аутентифицированный `gh`.

Без него Claude просто не сможет выполнить эти операции, даже если всё остальное настроено правильно. Установка и авторизация — одно из первых действий при настройке:

```bash
# Установка
sudo apt install gh

# Авторизация
gh auth login
```

`gh auth login` запустит интерактивный процесс: выбери GitHub.com, авторизацию через браузер, и следуй инструкциям.

## CLAUDE.md в проекте

`~/.claude/CLAUDE.md` — глобальные инструкции для Claude (RTK, хуки). Но для каждого конкретного проекта есть свой `CLAUDE.md` прямо в папке репозитория.

Это файл, который Claude читает при старте в этой папке. Туда пишут то, что Claude должен знать о проекте: стек, архитектурные решения, что нельзя трогать, как запускать тесты, соглашения по коду.

Без него Claude каждый раз разбирается в проекте заново. С ним — сразу работает в правильном контексте.

Минимальный пример:

```markdown
## Стек

Next.js 14, TypeScript, Prisma, PostgreSQL

## Запуск

npm run dev — локально
npm run test — тесты

## Важно

- Не трогать legacy/ без явной просьбы
- Все новые компоненты в src/components/
- Миграции только через Prisma, не руками
```

CLAUDE.md стоит добавить в репозиторий — тогда он работает для всей команды.

Рекомендуемый размер: до 200 строк, оптимально около 100. Файл читается целиком при каждом старте сессии и занимает место в контексте — чем короче, тем лучше.

## Как работают GSD и Superpowers вместе

Два инструмента закрывают разные этапы разработки и не дублируют друг друга.

**GSD (Get Shit Done)** — система планирования и управления контекстом. Главная проблема которую он решает — "context rot": по мере заполнения контекстного окна качество ответов деградирует. GSD структурирует работу так, чтобы Claude всегда работал в чистом контексте с актуальным планом. Начало работы над проектом: `/gsd-new-project`. Продолжение существующего: `/gsd-map-codebase` → `/gsd-new-project`.

**Superpowers** — скиллы для дисциплинированной разработки: мозгоштурм перед фичей, TDD, систематический дебаггинг, код-ревью. Работают как слэш-команды внутри Claude Code.

### Рабочий процесс

| Этап                  | Инструмент        | Что делает                                                                     |
| --------------------- | ----------------- | ------------------------------------------------------------------------------ |
| Планирование          | GSD               | Создаёт стабильную спецификацию и высокоуровневый план, структурирует контекст |
| Проектирование тестов | Superpowers + GSD | Генерирует падающие тесты на основе плана (TDD-подход)                         |
| Реализация            | Superpowers       | Пишет код, итеративно доводя до зелёных тестов                                 |
| Проверка              | Superpowers       | Юнит и E2E тесты, код-ревью (`/code-review`)                                   |
| Фиксация              | GSD               | Атомарный коммит с подтверждённым статусом (`/gsd-ship`)                       |

На существующем проекте GSD подключается через `/gsd-map-codebase`, а ревью текущей ветки можно запустить встроенным `/code-review`.

## Рекомендации по работе

Несколько принципов, которые экономят время и деньги независимо от проекта.

### Самое важное

**1. Твой главный советчик — сам Claude Code.** Лучший учитель, помощник и справочник по этому окружению — он сам. Не знаешь команду, настройку, почему что-то так устроено или как сделать X — просто спроси прямо в чате. Он знает свои возможности, твой сетап и эту документацию, объяснит и тут же поправит. Начинай с вопроса к нему, а не с гугла.

**2. Настрой интерфейс под себя — руками Клода.** Не подстраивайся под дефолты. Попроси Claude переписать `~/.bashrc`, хуки, статус-строку, цвета вкладок tmux так, как удобно тебе — он и настроит, и объяснит, что сделал. Этот репозиторий — пример одной такой настройки, а не догма; бери что нравится и меняй остальное.

**3. Не изобретай — ищи готовое.** Под почти любую задачу на GitHub уже есть готовый скилл, плагин или MCP-сервер. Сначала поиск (`/plugin`, маркетплейсы плагинов, репозитории со skills), и только потом — писать своё. Чужое решение часто закрывает 80% задачи за пять минут.

**4. Управляй контекстом, не тяни одну сессию бесконечно.** По мере заполнения окна качество ответов падает — это «context rot». `/clear` между несвязанными задачами, поглядывай на `ctx%` в статус-строке, а для длинной многоэтапной работы используй GSD — он именно для того, чтобы Claude всегда работал в чистом контексте с актуальным планом.

**5. Свой повторяющийся процесс оформляй скиллом или командой.** Если ты раз за разом диктуешь Claude один и тот же сценарий — опиши его один раз скиллом, и дальше вызывай командой вместо пересказа промпта. Так сделаны SEO-команды `/gist1…3` в этом сетапе.

**6. Планируй до кода.** Plan mode, brainstorming или `/gsd-plan-phase` перед тем, как пускать Claude писать. Полчаса на согласование плана дешевле, чем переделка половины фичи в неверном направлении.

**7. Требуй доказательств, а не «готово».** Пусть Claude запускает тесты и показывает вывод, открывает страницу в браузере, делает скриншот. «Сделал» без подтверждения — это «не сделал».

### Дальше

**8. Автоматизируй повторяющееся хуками, а не напоминаниями.** Всё в формате «каждый раз когда X — делай Y» (форматирование при сохранении, цвет вкладки, бип по завершении) — это хук в `settings.json`, а не строчка в памяти. Память Claude может проигнорировать, хук выполнит сам харнесс.

**9. Пусть Claude ведёт память и `CLAUDE.md` проекта.** Твои предпочтения и факты о проекте, которые не выводятся из кода, — в память и в `CLAUDE.md`, и обновлять их по ходу. Тогда не приходится объяснять одно и то же в каждой новой сессии.

**10. Дай инструменты, а не советы.** Подключай MCP-серверы под задачу — браузер (Playwright), GitHub, базу данных, Vercel. Разница как между «подскажи по телефону» и «пустить за компьютер»: с инструментами Claude делает, а не только советует.

**11. git как страховка.** Частые атомарные коммиты плюс worktrees для рискованного или параллельного — всегда можно откатиться. Claude работает смелее, когда есть откат, а ты ничего не теряешь.

**12. Настрой allowlist прав под себя.** Разреши рутину (`git`, `ls`, тесты) без подтверждений, чтобы не кликать каждый раз; `rm` и прочий деструктив оставь под вопросом. Команда `/fewer-permission-prompts` соберёт список кандидатов прямо из твоей истории.
