# Claude Code — Зачем это всё

Объяснение для тех, кто только знакомится с Claude Code и хочет понять почему окружение устроено именно так.

Инструкция по установке — в `setup.md`. Её можно отдать Claude, и он настроит всё сам.

---

## Что именно настраивается

`setup.md` делает следующее:

**Два алиаса** — `cc1` и `cc2` для быстрого запуска Claude без подтверждения каждого действия. `cc1` — личный аккаунт, `cc2` — рабочий, изолированный.

**RTK** — прокси для экономии токенов. После установки работает автоматически, ничего менять не нужно.

**Хуки в Claude Code:**

- при завершении ответа — звуковой сигнал и красная вкладка в tmux
- при записи файла — авто-форматирование (prettier, ruff, rustfmt, gofmt в зависимости от языка)
- перед каждой командой — RTK перехватывает и оптимизирует

**Статус-строка** внизу Claude Code — показывает какой аккаунт активен, модель, текущую ветку git (со звёздочкой если есть несохранённые изменения), заполненность контекста и rate limits.

**Список разрешений** — Claude может без вопросов выполнять стандартные команды (`git`, `docker`, `ls`, `python` и др.). `rm` намеренно исключён — требует подтверждения.

**tmux** — конфиг с удобными шорткатами, темой Catppuccin, сохранением сессий между перезагрузками (tmux-resurrect) и автосохранением каждые 15 минут.

## Claude Code — это не чат

Claude.ai (веб) — это диалог. Claude Code — это агент в терминале, который реально работает с твоей машиной: читает и пишет файлы, запускает команды, работает с git, docker, браузером. Разница примерно как между "попросить кого-то помочь по телефону" и "пустить человека за компьютер".

Это меняет сценарий использования: вместо "объясни мне как написать этот компонент" — "напиши этот компонент, запусти тесты, исправь ошибки, задеплой". Claude Code делает всё сам.

## Windows: почему WSL

Claude Code — Linux/macOS инструмент, на чистом Windows не работает. WSL (Windows Subsystem for Linux) даёт полноценный Linux прямо внутри Windows без отдельной виртуалки.

Из WSL можно:

- вызывать Windows-программы: `powershell.exe`, `explorer.exe`, `code.exe`
- обращаться к файлам Windows через `/mnt/c/...`

Из Windows можно обращаться к файлам WSL:

- в проводнике: адресная строка → `\\wsl$\Ubuntu` (или `\\wsl.localhost\Ubuntu`)
- папка домашнего каталога: `\\wsl$\Ubuntu\home\<username>`
- удобно добавить в "Быстрый доступ" проводника — правой кнопкой → "Закрепить на панели быстрого доступа"
- VS Code: запусти `code .` из WSL-терминала — откроется с автоматическим WSL Remote подключением

## Docker

В этом конфиге Docker используется как инструмент разработки — Claude работает с контейнерами через команды `docker *` и `docker-compose *`. Сам Claude Code запускается в WSL, не в контейнере.

Альтернативный сценарий — запустить Claude Code внутри Docker-контейнера для полной изоляции — здесь не используется.

## Почему tmux

Каждое отдельное окно WSL-терминала — это отдельный Windows-процесс со своим экземпляром Linux-окружения. При открытии 3–4 консолей память заканчивается быстро.

Tmux решает это радикально: **один процесс WSL**, внутри которого сколько угодно окон и панелей. Всё работает в одной сессии, память расходуется один раз.

Дополнительные преимущества:

- **Сессии не умирают** — закрыл терминал, открыл снова, всё на месте (`tmux attach`)
- **Несколько панелей** — Claude в одной, ты работаешь в другой, рядом логи
- **Цвет вкладки как уведомление** — вкладка tmux меняет цвет по состоянию агента: 🔵 синий — работает, 🔴 красный — закончил ход (твой ход), 🟡 жёлтый — ждёт пермишена. Видно, даже если смотришь на другое окно

**Запускать удобнее всего внутри VS Code** (WSL-remote режим): интегрированный
терминал авто-аттачится к сессии 0, и Ctrl+Click по путям из ответов Claude
открывает файлы вкладками рядом. Отдельный Windows Terminal не нужен.
Настройка — в `setup.md` §12.

## Почему RTK

Claude Code платный — каждый токен стоит денег. Стандартные инструменты выдают избыточный вывод: `git log` без параметров возвращает тысячи строк, Claude их читает, ты за них платишь.

RTK (Rust Token Killer) — прокси, который перехватывает команды Claude и заменяет их на компактные эквиваленты. `git log` → `rtk git log` (только нужные поля). Прозрачно, без изменения поведения, экономия 60–90% на типичных операциях.

Работает через хук `PreToolUse`: каждая bash-команда Claude автоматически проходит через RTK.

## Почему два аккаунта

Claude Code привязан к аккаунту Anthropic. Два аккаунта — два изолированных контекста:

- `claude1` (`cc1`) — личный, папка `~/.claude/`
- `claude2` (`cc2`) — рабочий, папка `~/.claude-work/`

У каждого свои плагины, своя история, свои лимиты. Конфиги не наследуются — каждый читает свой `settings.json` независимо.

## GitHub CLI (gh)

Claude Code умеет работать с GitHub — создавать PR, смотреть issues, проверять статус CI. Но для этого нужен аутентифицированный `gh`.

Без него Claude просто не сможет выполнить эти операции, даже если всё остальное настроено правильно. Установка и авторизация — одно из первых действий при настройке:

```bash
# Установка
sudo apt install gh

# Авторизация
gh auth login
```

`gh auth login` запустит интерактивный процесс: выбери GitHub.com, авторизацию через браузер, и следуй инструкциям.

## CLAUDE.md в проекте

`~/.claude/CLAUDE.md` — глобальные инструкции для Claude (RTK, хуки). Но для каждого конкретного проекта есть свой `CLAUDE.md` прямо в папке репозитория.

Это файл, который Claude читает при старте в этой папке. Туда пишут то, что Claude должен знать о проекте: стек, архитектурные решения, что нельзя трогать, как запускать тесты, соглашения по коду.

Без него Claude каждый раз разбирается в проекте заново. С ним — сразу работает в правильном контексте.

Минимальный пример:

```markdown
## Стек

Next.js 14, TypeScript, Prisma, PostgreSQL

## Запуск

npm run dev — локально
npm run test — тесты

## Важно

- Не трогать legacy/ без явной просьбы
- Все новые компоненты в src/components/
- Миграции только через Prisma, не руками
```

CLAUDE.md стоит добавить в репозиторий — тогда он работает для всей команды.

Рекомендуемый размер: до 200 строк, оптимально около 100. Файл читается целиком при каждом старте сессии и занимает место в контексте — чем короче, тем лучше.

## Как работают GSD, Superpowers и gstack вместе

Три инструмента закрывают разные этапы разработки и не дублируют друг друга.

**GSD (Get Shit Done)** — система планирования и управления контекстом. Главная проблема которую он решает — "context rot": по мере заполнения контекстного окна качество ответов деградирует. GSD структурирует работу так, чтобы Claude всегда работал в чистом контексте с актуальным планом. Начало работы над проектом: `/gsd-new-project`. Продолжение существующего: `/gsd-map-codebase` → `/gsd-new-project`.

**Superpowers** — скиллы для дисциплинированной разработки: мозгоштурм перед фичей, TDD, систематический дебаггинг, код-ревью. Работают как слэш-команды внутри Claude Code.

**gstack** — виртуальная инженерная команда от CEO YCombinator Garry Tan. Специализированные роли: CEO-ревью продукта (`/plan-ceo-review`), инженерное ревью (`/review`), QA с реальным браузером (`/qa`), деплой (`/ship`), аудит безопасности (`/cso`) и другие. Работает на любом существующем проекте без инициализации.

### Рабочий процесс

| Этап                  | Инструмент           | Что делает                                                                     |
| --------------------- | -------------------- | ------------------------------------------------------------------------------ |
| Планирование          | GSD                  | Создаёт стабильную спецификацию и высокоуровневый план, структурирует контекст |
| Проектирование тестов | Superpowers + GSD    | Генерирует падающие тесты на основе плана (TDD-подход)                         |
| Реализация            | Superpowers          | Пишет код, итеративно доводя до зелёных тестов                                 |
| Проверка              | Superpowers + gstack | Регрессионные, юнит и E2E тесты (`/qa`), ревью (`/review`)                     |
| Фиксация              | GSD                  | Атомарный коммит с подтверждённым статусом (`/gsd-ship`)                       |

Для существующего проекта gstack подключается сразу — `/retro` покажет историю, `/review` запустит ревью текущей ветки, `/health` — общее состояние.
